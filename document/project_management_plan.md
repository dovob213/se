## 1. 문서 개요

### 1.1 문서 목적

이 문서는 TeamFlow 프로젝트를 진행하기 위한 프로젝트 관리 계획서이다.
TeamFlow는 대학생 팀 프로젝트에서 자주 생기는 일정 누락, 역할 분담 혼란, 공지 전달 문제를 줄이기 위한 팀 프로젝트 일정 및 역할 관리 시스템이다.

프로젝트를 진행하다 보면 기능을 만드는 것뿐 아니라, 누가 어떤 작업을 맡는지, 일정은 어떻게 관리하는지, 문서는 어떻게 수정하는지 정리해두는 것도 중요하다고 생각했다. 그래서 이 문서에서는 TeamFlow 프로젝트의 범위, 일정, 역할, 품질 관리, 위험 관리, GitHub 관리 방식을 정리한다.

또한 조원에게 받은 리뷰를 반영하여 문서를 수정하고, 그 변경 이력을 GitHub에 남기는 것도 이 문서의 목적에 포함된다.

## 2. 프로젝트 범위 관리

### 2.1 프로젝트 목표

TeamFlow의 목표는 대학생 팀 프로젝트에서 필요한 일정 관리, 역할 분담, 공지 확인, 자료 공유, 진행률 확인 기능을 하나의 서비스에서 제공하는 것이다.

기존에는 팀 프로젝트를 할 때 카카오톡, 노션, 구글 드라이브, 개인 메모 등을 따로 사용하는 경우가 많았다. 그러다 보면 마감일을 놓치거나, 누가 어떤 일을 맡았는지 헷갈리는 일이 생긴다.

TeamFlow는 이런 문제를 줄이기 위해 팀 프로젝트에 필요한 핵심 기능을 단순하고 직관적으로 제공하는 것을 목표로 한다.

### 2.2 수행 범위

이번 프로젝트에서 수행할 범위는 다음과 같다.

1. 프로젝트 일정 관리 기능 설계
2. 팀원 역할 분담 기능 설계
3. 업무 진행률 확인 기능 설계
4. 공지사항 작성 및 확인 기능 설계
5. 자료 및 파일 첨부 기능 설계
6. 마감일 알림 기능 설계
7. 업무별 댓글 및 의견 공유 기능 설계
8. 프로젝트 문서 작성 및 GitHub 업로드
9. 조원 리뷰 확인 및 문서 수정

### 2.3 제외 범위

이번 과제에서는 실제 서비스를 완성하는 것보다 프로젝트 관리 계획을 세우는 것이 중심이다. 따라서 다음 내용은 범위에서 제외한다.

* 실제 웹 서비스 배포
* 실제 로그인 서버 구현
* 실제 파일 저장 서버 구축
* 모바일 앱 개발
* 외부 알림 서비스와의 실제 연동

## 3. 프로젝트 조직 및 역할

### 3.1 팀 구조

TeamFlow는 대학생 팀 프로젝트를 관리하는 시스템이기 때문에, 프로젝트의 기능을 한 번에 완성하기보다는 기능별로 나누어 점진적으로 개발하는 방식이 적합하다고 보았다.

이번 과제는 개인이 작성하는 문서가 중심이지만, 조원과 서로 리뷰를 주고받는 과정이 포함된다. 그래서 각자 독립적으로 문서를 작성하면서도, 리뷰 단계에서는 서로의 문서를 확인하는 분산형 구조에 가깝게 운영한다.

실제 TeamFlow 개발을 진행한다고 가정하면, 팀 내부에서는 기능별 담당자를 나누고 전체 방향은 PM이 관리하는 하이브리드 구조가 적합하다고 생각한다.

### 3.2 역할 정의

| 역할 | 담당 내용 |
|---|---|
| 프로젝트 관리자 | 전체 일정과 작업 흐름을 관리한다. |
| 기획 담당자 | TeamFlow의 주요 기능과 요구사항을 정리한다. |
| UI 담당자 | 사용자가 볼 화면 구조와 흐름을 정리한다. |
| 개발 담당자 | 일정 관리, 역할 분담, 공지 기능 등을 구현한다고 가정한다. |
| 품질 담당자 | 문서와 기능이 요구사항에 맞는지 점검한다. |
| 형상 관리자 | GitHub를 이용해 문서와 변경 이력을 관리한다. |

이번 과제에서는 위 역할을 모두 내가 중심이 되어 작성하지만, 리뷰 과정에서는 조원이 품질 담당자와 리뷰어 역할을 함께 수행한다고 볼 수 있다.

## 4. 일정 관리 계획

### 4.1 WBS

WBS는 TeamFlow 프로젝트를 진행한다고 가정하고, 실제로 확인 가능한 작업 단위로 나누었다.

| 작업 ID | 작업명 | 산출물 |
|---|---|---|
| WBS-01 | 프로젝트 정의 확인 | 프로젝트 정의서 |
| WBS-02 | 주요 기능 정리 | 기능 목록 |
| WBS-03 | 품질 요소 검토 | 품질 요소 추정서 |
| WBS-04 | 프로젝트 관리 계획서 작성 | 계획서 초안 |
| WBS-05 | 일정 및 역할 계획 정리 | WBS, 간트 차트 |
| WBS-06 | 기술 관리 및 품질 관리 작성 | 관리 계획 항목 |
| WBS-07 | 위험 관리 작성 | 위험 관리 목록 |
| WBS-08 | GitHub 업로드 | 커밋 기록 |
| WBS-09 | 조원 문서 리뷰 | Issue 또는 PR 리뷰 |
| WBS-10 | 리뷰 반영 | 수정 커밋 |
| WBS-11 | 최종 제출 정리 | 최종 문서 링크 |

각 작업의 의미는 다음과 같다.

* WBS-01: 기존에 작성한 프로젝트 정의서를 확인하고 TeamFlow의 목표를 다시 정리한다.
* WBS-02: 일정 관리, 역할 분담, 공지, 파일 공유, 댓글, 알림 기능을 정리한다.
* WBS-03: 사용성, 유지보수성, 신뢰성 등 TeamFlow에 필요한 품질 요소를 확인한다.
* WBS-04: 프로젝트 관리 계획서의 전체 틀을 작성한다.
* WBS-05: 작업을 WBS로 나누고 간트 차트 형태로 일정을 정리한다.
* WBS-06: GitHub 관리 방식, 커밋 규칙, 품질 점검 방법을 작성한다.
* WBS-07: 일정 지연, 요구사항 변경, Git 사용 실수 같은 위험을 정리한다.
* WBS-08: 작성한 문서를 GitHub 저장소에 업로드한다.
* WBS-09: 조원의 문서를 읽고 간단한 리뷰를 남긴다.
* WBS-10: 내가 받은 리뷰를 확인하고 문서에 반영한다.
* WBS-11: 최종 문서 링크와 제출 상태를 확인한다.

## 5. 기술 관리 계획

### 5.1 버전 관리

문서의 변경 이력은 GitHub를 통해 관리한다.
프로젝트 정의서, 품질 요소 추정서, 프로젝트 관리 계획서를 같은 저장소에서 관리하여 과제 문서의 흐름을 한눈에 볼 수 있도록 한다.

문서는 다음 경로에 저장한다.

* doc/definition_document.md
* doc/quality_factors.md
* doc/project_management_plan.md

### 5.2 브랜치 전략

브랜치는 복잡하게 운영하지 않고, 과제 수행에 필요한 최소한의 흐름만 사용한다.

* main: 최종 제출 가능한 문서를 보관하는 브랜치
* assignment2/project-management-plan: 프로젝트 관리 계획서를 작성하는 작업 브랜치
* assignment2/reflect-review: 조원 리뷰를 반영하는 작업 브랜치

문서 수정이 크지 않은 경우 main에 바로 반영할 수도 있지만, 리뷰 반영이나 파일명 변경처럼 의미 있는 수정은 별도 커밋으로 남긴다.

### 5.3 커밋 메시지 규칙

커밋 메시지는 나중에 이력을 봤을 때 어떤 작업인지 알 수 있게 작성한다.

* docs: add project management plan
* docs: update wbs and schedule
* docs: reflect peer review comments
* docs: organize project document links

### 5.4 변경 관리

TeamFlow의 기능이나 문서 구성에 수정이 필요할 경우, 먼저 어떤 부분을 바꿀지 확인한 뒤 문서를 수정한다.
특히 조원 리뷰를 반영한 내용은 별도 커밋으로 남겨서, 리뷰 전 문서와 리뷰 후 문서의 차이를 확인할 수 있도록 한다.

## 6. 품질 관리 계획

### 6.1 품질 목표

TeamFlow 프로젝트 문서의 품질 목표는 다음과 같다.

* TeamFlow의 목적과 핵심 기능이 문서에서 분명하게 드러나야 한다.
* 일정 관리, 역할 관리, 품질 관리, 위험 관리 항목이 빠지지 않아야 한다.
* 프로젝트 정의서, 품질 요소 추정서, 프로젝트 관리 계획서의 내용이 서로 어긋나지 않아야 한다.
* GitHub에 문서 작성과 수정 이력이 남아야 한다.
* 조원 리뷰를 실제로 반영한 흔적이 있어야 한다.
* 맞춤법이나 오타 때문에 문서의 가독성이 떨어지지 않도록 한다.

### 6.2 품질 관리 방법

이번 프로젝트에서는 다음 방식으로 품질을 관리한다.

* 자체 점검: 제출 전에 체크리스트를 보며 누락된 항목이 없는지 확인한다.
* 동료 리뷰: 조원이 남긴 의견을 통해 부족한 부분을 확인한다.
* 형상 관리: Git commit을 이용해 문서의 변경 내역을 추적한다.
* 내용 일관성 확인: 프로젝트 정의서의 기능과 프로젝트 관리 계획서의 범위가 맞는지 확인한다.
* 최종 확인: GitHub main 브랜치에서 문서 링크가 정상적으로 열리는지 확인한다.

## 7. 위험 관리 계획

이번 프로젝트에서 예상되는 위험은 일정 지연, 요구사항 변경, Git 사용 실수, 문서 간 내용 불일치와 관련되어 있다.

### R-01. 일정 지연

* 영향: 문서 작성과 리뷰 반영이 늦어져 제출 시간이 부족해질 수 있다.
* 대응: WBS 기준으로 작업을 나누고, 초안을 먼저 작성한 뒤 수정한다.

### R-02. 요구사항 변경

* 영향: TeamFlow의 기능 범위가 바뀌면 계획서의 일정과 역할도 함께 수정해야 한다.
* 대응: 변경된 요구사항을 먼저 정리하고, 필요한 경우 범위와 WBS를 함께 수정한다.

### R-03. Git 사용 오류

* 영향: 문서가 정상적으로 업로드되지 않거나 작업 내용이 꼬일 수 있다.
* 대응: 작업 전후로 git status, git log, git remote -v를 확인한다.

### R-04. 조원 리뷰 누락

* 영향: 과제 조건을 충족하지 못할 수 있다.
* 대응: 조원에게 문서 링크를 공유받고, Issue 또는 PR로 리뷰 기록을 남긴다.

### R-05. 받은 리뷰 반영 누락

* 영향: 리뷰를 받았지만 개선 과정이 보이지 않을 수 있다.
* 대응: 리뷰 내용을 체크리스트처럼 확인하고, 반영한 내용을 별도 커밋으로 남긴다.

### R-06. 문서 간 내용 불일치

* 영향: 프로젝트 정의서에는 있는 기능이 관리 계획서에는 빠지거나, 품질 요소 추정서와 다른 내용이 생길 수 있다.
* 대응: 최종 제출 전에 프로젝트 정의서, 품질 요소 추정서, 프로젝트 관리 계획서를 함께 확인한다.

## 8. 의사소통 및 리뷰 계획

### 8.1 의사소통 방식

과제 수행 중 의사소통은 GitHub를 중심으로 진행한다.

* GitHub Issue: 문서에 대한 개선 의견을 남길 때 사용한다.
* Pull Request Review: 직접 수정 제안을 하거나 변경 내용을 검토할 때 사용한다.
* 메신저 또는 대면: 리뷰 요청, 마감 일정 공유처럼 간단한 연락에 사용한다.

### 8.2 리뷰 작성 기준

리뷰는 단순히 좋다고만 적기보다, 상대방이 바로 수정할 수 있는 정도로 작성한다.
너무 깊게 들어가기보다는 문서 양식, 일정 표현, 빠진 항목, 오타처럼 고치기 쉬운 부분을 중심으로 확인한다.

리뷰 예시는 다음과 같다.

좋았던 점:
프로젝트 목표와 기능이 잘 정리되어 있어 어떤 시스템을 만들려는지 이해하기 쉬웠습니다.

개선하면 좋을 점:
일정 계획이 글로만 작성되어 있어, 표나 간트 차트 형태로 정리하면 더 한눈에 보기 좋을 것 같습니다.
