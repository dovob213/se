## 2. 프로젝트 범위 관리

이번 과제의 목표는 프로젝트 관리 계획서의 구성 요소를 이해하고, GitHub를 이용해 문서 작성과 리뷰, 수정 과정을 남기는 것이다.
따라서 결과물 자체도 중요하지만, 작성 과정과 수정 이력이 GitHub에 남는 것도 중요한 수행 범위에 포함된다.

### 2.2 수행 범위

이번 과제에서 수행할 작업은 다음과 같다.

1. ch05 프로젝트 관리 강의자료 읽기

3. 작성한 문서를 GitHub 저장소에 업로드하기
4. 조원이 작성한 문서를 읽고 Issue 또는 PR로 리뷰 남기기
5. 내가 받은 리뷰를 확인하고 문서에 반영하기
6. 최종 수정본을 다시 GitHub에 반영하기

* 실제 소프트웨어 구현

## 3. 프로젝트 조직 및 역할

### 3.1 팀 구조

이번 과제는 각자 자신의 문서를 작성하지만, 조원끼리 서로 리뷰를 주고받아야 한다.
그래서 완전한 개인 과제라기보다는, 각자가 독립적으로 작업하면서도 리뷰 단계에서는 협업하는 구조라고 볼 수 있다.

따라서 팀 구조는 분산형 팀 구조에 가깝게 운영한다. 각 조원은 자신의 문서에 대한 책임을 가지고, 다른 조원의 문서에 대해서는 리뷰어 역할을 수행한다.

### 3.2 역할 정의

| 역할     | 담당 내용                                  |
| ------ |----------------------------------------|
| 작성자    | 자신의 프로젝트 관리 계획서를 작성하고 GitHub에 업로드한다.   |
| 리뷰어    | 조원의 문서를 읽고 Issue 또는 PR로 개선 의견을 남긴다.    |
| 문서 관리자 | 받은 리뷰를 확인하고 문서를 수정한다.                  |
| 형상 관리자 | Git branch, commit, push, PR 흐름을 관리한다. |

실제 과제에서는 위 역할을 모두 내가 수행하지만, 조원과의 리뷰 과정에서는 작성자와 리뷰어 역할이 서로 나뉘게 된다.

## 4. 일정 관리 계획

### 4.1 WBS

WBS는 작업을 너무 크게 잡지 않고, 실제로 수행하고 확인할 수 있는 단위로 나누었다.

| 작업 ID  | 작업명        | 산출물            |
| ------ | ---------- | -------------- |
| WBS-01 | 강의자료 확인    | 학습 메모          |
| WBS-02 | 계획서 초안 작성  | 초안 문서          |
| WBS-03 | GitHub 업로드 | 커밋 기록          |
| WBS-04 | 조원 문서 리뷰   | Issue 또는 PR 리뷰 |
| WBS-05 | 리뷰 반영      | 수정 커밋          |
| WBS-06 | 최종 제출 정리   | 최종 문서 링크       |

각 작업의 의미는 다음과 같다.

* **WBS-01**: ch05 강의자료에서 프로젝트 관리 계획서에 들어가야 할 항목을 확인한다.
* **WBS-02**: 문서 목적, 범위, 일정, 역할, 품질 관리, 위험 관리 내용을 작성한다.
* **WBS-03**: 작성한 문서를 GitHub 저장소의 'document/' 폴더에 올린다.
* **WBS-04**: 조원의 문서를 읽고 개선할 점이나 좋았던 점을 리뷰로 남긴다.
* **WBS-05**: 내가 받은 리뷰를 반영해 문서를 수정한다.
* **WBS-06**: 최종 문서가 main 브랜치에 잘 반영되었는지 확인하고 제출 링크를 정리한다.

## 5. 기술 관리 계획

### 5.1 버전 관리

문서의 변경 이력은 GitHub를 통해 관리한다.
초안 작성, 리뷰 반영, 최종 제출 정리처럼 의미 있는 변경이 있을 때마다 커밋을 남긴다.

문서는 다음 경로에 저장한다.

```text
document/assignment2_project_management_plan.md
```

### 5.2 브랜치 전략

브랜치는 복잡하게 운영하지 않고, 과제 수행에 필요한 최소한의 흐름만 사용한다.

* main: 최종 제출 가능한 문서를 보관하는 브랜치
* assignment2/project-management-plan: 과제2 문서를 작성하는 작업 브랜치
* assignment2/reflect-review: 조원 리뷰를 반영하는 작업 브랜치

문서 초안을 바로 main에 올릴 수도 있지만, Pull Request 흐름을 연습하기 위해 작업 브랜치를 사용한다.

### 5.3 커밋 메시지 규칙

커밋 메시지는 나중에 이력을 봤을 때 어떤 작업인지 알 수 있게 작성한다.

```text
docs: add assignment2 project management plan
docs: update schedule and wbs
docs: reflect peer review comments
chore: organize document folder
```

### 5.4 변경 관리

리뷰를 통해 수정이 필요한 내용이 생기면 바로 문서를 고치기보다, 먼저 어떤 의견을 반영할지 확인한다.
반영한 내용은 별도 커밋으로 남겨서 '리뷰 전 문서'와 '리뷰 후 문서'의 차이를 확인할 수 있도록 한다.

## 7. 위험 관리 계획

이번 과제에서 예상되는 위험은 대부분 일정 지연, Git 사용 실수, 리뷰 누락과 관련되어 있다.

### R-01. 문서 작성 지연

* 영향: 제출 시간이 부족해져 문서 품질이 낮아질 수 있다.
* 대응: WBS 기준으로 작업을 나누고, 초안을 먼저 작성한 뒤 수정한다.

### R-02. Git 사용 오류

* 영향: 문서가 정상적으로 업로드되지 않거나 작업 내용이 꼬일 수 있다.
* 대응: 작업 전후로 `git status`, `git log`, `git remote -v`를 확인한다.

### R-03. 조원 리뷰 누락

* 영향: 과제 조건을 충족하지 못할 수 있다.
* 대응: 조원에게 문서 링크를 공유받고, Issue 또는 PR로 리뷰 기록을 남긴다.

### R-04. 받은 리뷰 반영 누락

* 영향: 리뷰를 받았지만 개선 과정이 보이지 않을 수 있다.
* 대응: 리뷰 내용을 체크리스트처럼 확인하고, 반영한 내용을 별도 커밋으로 남긴다.

### R-05. 문서 양식 깨짐

* 영향: GitHub에서 문서가 읽기 어렵게 보일 수 있다.
* 대응: 긴 표는 줄이고, 목록과 코드 블록, Mermaid 차트를 함께 사용한다.

## 8. 의사소통 및 리뷰 계획

### 8.1 의사소통 방식

과제 수행 중 의사소통은 GitHub를 중심으로 진행한다.

* **GitHub Issue**: 문서에 대한 개선 의견을 남길 때 사용한다.
* **Pull Request Review**: 직접 수정 제안을 하거나 변경 내용을 검토할 때 사용한다.
* **메신저 또는 대면**: 리뷰 요청, 마감 일정 공유처럼 간단한 연락에 사용한다.

### 8.2 리뷰 작성 기준

리뷰는 단순히 “좋습니다”라고 끝내지 않고, 상대방이 실제로 수정할 수 있게 작성한다.
좋았던 점과 개선하면 좋을 점을 함께 적는 방식으로 진행한다.

리뷰 예시는 다음과 같다.

```text
좋았던 점:
문서 목적과 수행 범위가 명확해서 과제의 방향을 이해하기 쉬웠습니다.

개선하면 좋을 점:
WBS에는 문서 작성뿐 아니라 조원 리뷰와 리뷰 반영 작업도 포함하면 더 좋을 것 같습니다.
또한 품질 관리 항목에서 체크리스트를 추가하면 최종 제출 전에 확인하기 쉬울 것 같습니다.
```
